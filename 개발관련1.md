# triple (PRO) - 고성능 최적화 에디터 가이드 (README2.md)

**triple (PRO)** 버전은 파이썬 백엔드 환경을 동일하게 사용하면서, 사용자가 입력하고 셀을 조작할 때 발생하는 프론트엔드 브라우저(WebView2) 단의 병목을 완벽히 해결하여 **체감 반응 속도와 렌더링 성능을 극대화한 고성능 빌드**입니다.

---

## 🛠️ 적용된 초고속 렌더링 & 최적화 핵심 기술

### 1) 레이아웃 스래싱(Layout Thrashing)의 근본적 제거
* **기존 문제점**: 에디터 셀의 높이를 맞출 때, 각 셀마다 순차적으로 높이를 `auto`로 설정하고 `scrollHeight`를 읽어온 뒤 최대 높이를 지정했습니다. 이 과정에서 브라우저가 매번 화면을 강제로 재배치하여 텍스트 분량이 길어질수록 타이핑 및 렌더링이 극도로 버벅였습니다.
* **PRO 최적화 (Batch Read/Write 기법)**:
  * **1단계 (일괄 쓰기)**: 모든 타겟 셀의 높이를 `'auto'`로 먼저 한 번에 변경합니다.
  * **2단계 (일괄 읽기)**: 모든 셀의 `scrollHeight` 값을 한 번에 수집합니다. (브라우저가 화면 구조를 단 1번만 계산하므로 CPU/GPU 소모가 비약적으로 감소합니다.)
  * **3단계 (일괄 쓰기)**: 각 라인별 최댓값(Max Height)을 구해 한꺼번에 반영합니다.

### 2) DOM 인플레이스 재사용 (In-place DOM Management)
* **기존 문제점**: 엔터를 쳐서 셀을 나누거나 백스페이스로 병합할 때 에디터 내부의 모든 셀 HTML을 완전히 날리고 처음부터 다시 100개 이상의 돔 행을 렌더링(`container.innerHTML = ''`)하여 극심한 딜레이와 스크롤 튐이 발생했습니다.
* **PRO 최적화**:
  * 전체 셀을 다시 렌더링하지 않고, 쪼개지거나 합쳐지는 **특정 위치의 단 하나의 셀 DOM만 생성하여 `insertBefore` 하거나 `remove()`** 합니다.
  * 돔이 새로 꽂힌 후, 가볍게 루프를 돌며 `data-block-index` 속성과 채색 세그먼트의 클래스명만 새로 인덱싱(`reindexAllBlocks()`)해 줍니다. 텍스트가 아무리 길어져도 버벅임이 전혀 느껴지지 않는 극상의 조작감을 체감할 수 있습니다.

### 3) requestAnimationFrame 렌더링 프레임 동기화
* 사용자가 에디터에 타이핑하는 매 순간(`input` 이벤트) 즉각적으로 무거운 레이아웃 연산이 돌지 않도록, 브라우저가 다음 화면 프레임을 그리는 시점에 맞춰 일괄 연산이 실행되도록 `requestAnimationFrame` 큐에 연산을 예약하여 스레드 정체를 방지했습니다.

### 4) 그래픽 하드웨어 GPU 가속
* 개별 셀 경계선 세그먼트(`.block-border-segment`)의 호버 및 투명도 전환 연산에 `will-change: opacity, box-shadow;` 속성을 부여하여 윈도우 그래픽 카드(GPU)의 가속 렌더링을 유도하고 CPU의 연산 부담을 덜어주었습니다.
